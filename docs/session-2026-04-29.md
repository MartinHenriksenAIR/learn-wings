### BUG-01 · Page reload causes 404 on all routes
**File added:** `staticwebapp.config.json` (repo root)

Azure Static Web Apps treats every URL as a server-side file request. Without a fallback config, reloading on any React Router route returns a real 404 from Azure. The fix adds a navigation fallback that tells Azure to serve `index.html` for all unmatched paths, while excluding static assets so they still serve directly. React Router then reads the URL from the browser address bar and renders the correct view. The user is returned to exactly the view they were on before reloading — provided their auth session is still valid.

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/assets/*", "/*.{css,js,ico,png,jpg,svg,woff,woff2}"]
  }
}
```

### BUG-04 · Course list sorting breaks when toggling publish state
**File modified:** `src/pages/platform-admin/CoursesManager.tsx`

Courses were intermixed (published and drafts in one unsorted list) and toggling publish state caused a full re-fetch that re-sorted the list, moving a course to the end. Fixed by:
- Splitting courses into two stable arrays sorted by `created_at` descending: `publishedFilteredCourses` and `draftFilteredCourses`
- Rendering a visual section divider with label ("Published" / "Drafts") and a count badge between the two groups
- Replacing the full re-fetch on toggle with an optimistic local state update — the course moves to the correct group instantly, and reverts if the DB write fails

Sort is by `created_at` (creation date). Sorting by `updated_at` was considered but ruled out — the platform is not expected to have enough courses to make editorial sort order meaningful.

### BUG-06 · Global Analytics default view ("All Organizations") shows nothing
**Files modified:** `src/pages/org-admin/OrgAnalytics.tsx`, `src/components/org-admin/analytics/TeamPerformanceTab.tsx`, `src/components/org-admin/analytics/CourseProgressTab.tsx`

**Important architectural note:** Both the Org Admin analytics route (`/app/admin/analytics`) and the Platform Admin global analytics route (`/app/admin/analytics/global`) use the same `OrgAnalytics` component. The component detects which view it is in via `location.pathname === '/app/admin/analytics/global'` (`isGlobalView` flag). The Org Admin sidebar only links to their own org's analytics — they never see the "All Organizations" option.

**Root cause:** Data fetches were wrapped in `if (orgFilter)` guards. When "All Organizations" was selected (orgFilter = null/undefined), nothing was fetched and the tabs rendered empty.

**Fix:**
- Removed the `if (orgFilter)` guard from the data fetch in `OrgAnalytics.tsx`
- `TeamPerformanceTab.tsx`: removed early return when `orgId` is falsy; now fetches all user stats across orgs with deduplication
- `CourseProgressTab.tsx`: removed `{effectiveOrgId ? <content> : <placeholder>}` guard; now fetches all courses and enrollment counts when no org is selected
- `orgId` prop type updated from `string` to `string | undefined` on both child components

**Outstanding issue — logged as FEAT-11:** The current fix renders a flat list of all learners across all orgs, which is unworkable at scale. The intended design (confirmed during review) is:
- Level 1: One row per organization (aggregated — org name, learner count, completion rate, avg score)
- Level 2: Expandable per org to show one row per learner in that org

This requires a separate aggregation query and a different table structure in the global view. Logged in `docs/backlog-features.md` as FEAT-11.

### BUG-07 · "Test Connection" button enabled with empty required fields
**File modified:** `src/pages/platform-admin/PlatformSettings.tsx`

The Test Connection button in the Email settings tab was always enabled. Clicking it with empty fields caused a confusing silent failure or error. Fixed by adding a computed boolean `isSmtpFormValid` that checks all six required fields are non-empty before enabling the button.

**Required fields (confirmed during review):**
1. `smtp_host`

3. `smtp_username`

Sender Name and Sender Email were added during the review walkthrough — the initial fix only included the four SMTP fields. Button receives `disabled={testingSmtp || !isSmtpFormValid}` and Tailwind disabled styles (`disabled:cursor-not-allowed disabled:opacity-50`).

### BUG-08 · Reporting community content does not work
**File added:** `supabase/migrations/20260429121759_fix_community_reports_rls.sql`

**No application code was changed.** Root cause was a missing RLS SELECT policy on the `community_reports` table.

**Failure chain:**
1. User clicks Report on a post/comment
2. `createReport()` in `community-api.ts` inserts the report row (succeeds)
3. Immediately after INSERT, it chains `.select().single()` to return the new row
4. RLS blocks the SELECT — the reporter had no read permission on the table
5. PostgREST returns a PGRST116 "no rows" error
6. The function throws before returning, triggering `onError` instead of `onSuccess`
7. The `ReportDialog.handleSubmit` has no catch block, so the error is swallowed entirely
8. User sees nothing — no success toast, no error toast

**Fix:** Added one RLS SELECT policy:
```sql
CREATE POLICY "Reporters can view their own reports"
  ON public.community_reports FOR SELECT
  USING (reporter_user_id = auth.uid());
```

**⚠️ Production status:** This migration file exists in the repo but has NOT been automatically applied to the live Supabase instance (`cairuxpyfshugwjrrqha`). Migration files in this project are not auto-applied on deploy. Reporting is still broken in production until one of the following is done:
- **Option A:** Paste and run the SQL in the Supabase Dashboard → SQL Editor
- **Option B:** Run `supabase db push` via the CLI

This exposes a broader risk: there is no automated migration pipeline. Every migration requires manual application. A GitHub Actions step running `supabase db push` post-deploy would eliminate this risk.

## 6. Supabase Architecture — Confirmed Active Usage

A full dependency audit was conducted. Supabase is the sole backend for the entire application.

### Scale of usage
- **41 database tables** with full RLS policies and multi-org tenant isolation
- **10 Edge Functions** (grade-quiz, generate-certificate, send-invitation-email, azure-upload-url, azure-view-url, azure-delete-blob, azure-document-upload-url, test-smtp-connection, delete-user, generate-compliance-report)
- **2 storage buckets** (lms-assets, org-logos)
- **44 migration files** in `supabase/migrations/`
- **40+ source files** importing the Supabase client
- **Project ID:** `cairuxpyfshugwjrrqha`

### Auth flow
```
Login → signInWithPassword → onAuthStateChange → profiles (SELECT) → org_memberships (SELECT)
```

### Key RLS helper functions (enforced at DB level)
- `is_platform_admin()` — checks platform admin status
- `is_org_admin(check_org_id)` — checks org admin for specific org
- `is_org_member(check_org_id)` — checks active membership
- `current_org_ids_for_user()` — returns all orgs user belongs to

### All Edge Functions have `verify_jwt = false`
They implement custom auth logic — manually validating Bearer tokens and checking admin status via service role queries. This is intentional but means auth is enforced in application code rather than at the Supabase gateway level.

### Migration pipeline risk
There is no automated migration pipeline. Migrations committed to the repo must be manually applied to the live database. This is a deployment risk: code and database schema can drift silently. Recommendation: add a `supabase db push` step to the GitHub Actions deployment workflow.
