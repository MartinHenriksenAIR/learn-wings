## Context

The platform stores LMS video and document assets, and email logo assets. Originally stored in Supabase Storage. The migration moves all binary assets to Azure. Access control for protected assets (LMS content) must be per-request and time-limited; public assets (email logo) must be permanently accessible.

## Decision

Use Azure Blob Storage (storage account: staieducationmigration) for all file assets. Protected content (lms-videos, lms-documents containers) is accessed via short-lived SAS tokens generated by Azure Functions (azure-upload-url, azure-view-url, azure-delete-blob). Public content (email-assets container) uses blob-level public access. Don't use Supabase Storage. Don't generate SAS tokens client-side.

## Consequences

Positive: SAS tokens expire (default 1 hour), no CORS rules needed for SAS-based access, existing containers (lms-videos, lms-documents) already provisioned. Negative: email-assets container must be created and logo uploaded before cutover; SAS token generation requires function call per asset access.

## Alternatives

1. Keep Supabase Storage — rejected: contradicts migration goal. 2. Azure CDN with token auth — rejected: additional complexity not justified at current scale. 3. S3-compatible storage (Backblaze, Cloudflare R2) — rejected: not in Azure ecosystem, breaks Key Vault secret pattern. 4. Azure Files — rejected: designed for SMB/NFS mount, not HTTP blob access.
